<TLDR>
  Over 3.5 years on Pleo’s design systems team, I worked across design, code,
  and documentation to strengthen the system’s foundations. I led the
  introduction of semantic tokens, established accessibility improvements,
  maintained and governed Figma libraries, and contributed directly to the
  codebase. This work helped align designers and engineers around a shared
  language and supported safer, more scalable changes across Pleo’s products.
</TLDR>

## Context

I spent 3.5 years on Pleo’s design systems team, working in a cross-functional team of two designers and three engineers. The design system supported both web and mobile products and was used daily by 20+ product designers and 60+ front-end engineers across the organisation.

The team’s responsibility went beyond building and maintaining components. We owned the foundations that made the system coherent, accessible, and scalable as the product and organisation evolved. This meant working across design, code, documentation, and process, and often translating broad or ambiguous needs into clear system-level solutions.

My work spanned several areas, including semantic tokens, accessibility foundations, Figma libraries, code contributions, documentation, and internal workflows. The sections below break these down in more detail.

## Semantic tokens

I led the introduction of semantic tokens to Pleo’s design system, defining the naming system, building Figma token libraries, writing documentation, running onboarding workshops, and migrating components in both Figma and code.

This delivered several key benefits:

<HighlightList>
  <Highlight title="Consistency" icon="consistency">
    Moving from core to semantic tokens improved visual alignment across the
    design system and product.
  </Highlight>
  <Highlight title="Efficiency" icon="speed">
    Designers could make faster, more confident decisions by selecting tokens
    based on meaning (e.g. `color-background-warning`).
  </Highlight>
  <Highlight title="Scalability" icon="scale">
    Product-wide visual changes could be made centrally and rolled out safely.
  </Highlight>
  <Highlight title="Theming" icon="theme">
    The semantic setup enabled both dark mode and product white-labelling.
  </Highlight>
  <Highlight title="Collaboration" icon="collaboration">
    Semantic tokens gave designers and engineers a shared, intent-based
    language, improving handoff quality and reducing implementation ambiguity.
  </Highlight>
</HighlightList>

## Accessibility improvements

I led and delivered accessibility improvements across the design system, spanning design, code, documentation, and process.

<HighlightList>
  <Highlight title="Accessible focus states" icon="focus">
    Designed, documented, and implemented a new focus ring that balanced brand
    expression with WCAG requirements.
  </Highlight>
  <Highlight title="Form accessibility foundations" icon="form">
    Defined specs and wrote documentation to ensure form components exposed
    accessible names and descriptions, and communicated required and invalid
    states correctly to screen readers.
  </Highlight>
  <Highlight title="Colour accessibility guidance" icon="colour">
    Authored WCAG-compliant colour guidelines and built a contrast matrix of
    semantic token pairings to support automated contrast testing.
  </Highlight>
  <Highlight title="Component-level improvements" icon="build">
    Independently identified and resolved accessibility issues in component
    code.
  </Highlight>
  <Highlight title="External audit triage" icon="audit">
    Triaged 100+ external audit findings, distinguishing system flaws from
    misuse, and routing prioritized fixes into the design system backlog.
  </Highlight>
  <Highlight title="More resilient sizing" icon="textScaling">
    Helped drive the migration from `px` to `rem` units to support user-defined
    text scaling and accessibility preferences.
  </Highlight>
</HighlightList>

## Figma component checklist

I created a checklist for design system components and built it as a Figma component. Because it lived alongside the work, designers could reference it without switching context, reducing cognitive load and improving adherence.

The checklist improved:

<HighlightList>
  <Highlight title="Quality" icon="quality">
    Expectations became explicit and consistent. States, variants, token usage,
    naming, and documentation were no longer missed or applied inconsistently,
    resulting in more complete and predictable components and less rework.
  </Highlight>
  <Highlight title="Speed" icon="speed">
    Designers spent less time recalling steps, checking guidelines, or asking
    questions. Reviews were shorter and more focused, reducing back-and-forth
    and helping components move faster through the system.
  </Highlight>
  <Highlight title="Scalability" icon="scale">
    Standards were embedded directly in the workflow, reducing reliance on
    tribal knowledge and enabling contributors to add components with
    confidence. Updating standards was easier, as changes could be applied
    centrally.
  </Highlight>
</HighlightList>

My work focused on:

This documentation reduced ambiguity during handoff and helped teams use the design system more confidently and consistently.
